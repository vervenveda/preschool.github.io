# Security Policy

## 1. Purpose

This security policy applies to **Sunny ABCs & 123s — Storytime** and all related Khaemenes Academy, Verve N Veda, and Crechè Pre School applications, tools, source files, documentation, downloadable resources, and future versions.

This policy is intended to:

- Protect children, families, educators, and other users
- Protect locally stored learner information and progress
- Encourage responsible reporting of security concerns
- Prevent unauthorized access, modification, or exploitation
- Preserve the integrity of official project files and releases
- Establish clear expectations for supported and unsupported use

This document does not grant permission to copy, modify, redistribute, reverse engineer, scrape, test, or exploit the Project beyond what is expressly permitted by the applicable proprietary license.

## 5. Security and Privacy Design

Official applications are designed, where practical, around the following principles:

- Single-file HTML, CSS, and JavaScript architecture
- No required user account
- No advertising
- No analytics
- No intentional behavioral tracking
- No unnecessary external scripts
- No unnecessary network requests
- Local browser storage for progress and preferences
- User-initiated downloads and exports
- Minimal collection of personal information
- Clear separation between official content and external links
- Private, offline-friendly operation

Not every browser, operating system, or archived version behaves identically. Users remain responsible for maintaining a secure device and browser environment.

## 6. Locally Stored Information

Some applications may use browser local storage, session storage, IndexedDB, cache storage, or downloadable files to retain information such as:

- Learner progress
- Preferences
- Stars or rewards
- Visited pages
- Favorite pages
- Locally created artwork
- Activity results
- User-entered artwork titles
- Optional names entered for certificates or reports

This information ordinarily remains on the user’s device unless the user independently exports, uploads, shares, synchronizes, or transmits it elsewhere.

Local information may be lost when:

- Browser data is cleared
- Private-browsing mode is used
- A browser removes stored data
- A device is reset or replaced
- Storage limits are exceeded
- An operating system or browser update interferes with storage
- The user changes browsers or devices
- A local application file is moved, renamed, or deleted

The Project does not guarantee permanent retention or recovery of locally stored information.

## 9. Reporting a Security Concern

Security concerns should be reported privately and responsibly.

A useful report should include:

- Name of the affected application
- File name or page address
- Version or release date
- Browser and operating system
- Description of the concern
- Steps needed to reproduce it
- Expected behavior
- Actual behavior
- Screenshots or recordings when helpful
- Whether personal information may have been exposed
- Whether active exploitation is suspected
- Suggested mitigation, when known

Do not include unnecessary personal information, student records, passwords, private keys, access tokens, or confidential documents in a report.

Use the official contact method listed on an authorized Khaemenes Academy or Verve N Veda website or repository.

## 12. Out-of-Scope Reports

The following matters are generally outside the security scope unless they create a meaningful security or privacy risk:

- Typographical errors
- Curriculum disagreements
- Visual preferences
- Missing features
- Unsupported older browsers
- Device-specific speech-voice differences
- Speech-synthesis quality
- Loss of data after browser storage is deliberately cleared
- Expected private-browsing behavior
- Broken external websites not controlled by the Project
- Third-party vulnerabilities with no Project-specific impact
- Reports requiring physical access to an already unlocked device
- Problems found only in unauthorized copies
- Theoretical concerns without practical impact

Copyright infringement, impersonation, unauthorized redistribution, license violations, and misuse of branding should be reported as intellectual-property or legal concerns rather than software vulnerabilities.

---

## 13. Security Review Process

After receiving a credible report, the Project owner may:

1. Confirm receipt when practical.
2. Review the affected version.
3. Attempt to reproduce the concern.
4. Assess the effect on children, families, educators, and local data.
5. Determine whether the issue affects an official release.
6. Develop a correction or mitigation when appropriate.
7. Publish an updated official version.
8. Document significant changes.
9. Notify affected users when a practical notification channel exists.

Response times may vary depending on severity, complexity, available resources, and whether the issue affects a current official release.

No guaranteed response time, repair time, payment, reward, attribution, or public acknowledgment is promised.

## 14. Severity Guidance

### Critical

Examples may include:

- Exposure of sensitive child or learner information
- Unauthorized remote code execution in an official release
- Malicious code distributed through an official channel
- Compromise of an official repository or administrative account
- A vulnerability allowing widespread unauthorized access

### High

Examples may include:

- Persistent script injection in an official hosted application
- Significant unauthorized modification of official files
- Exposure of locally stored information without meaningful user action
- A practical method for impersonating an official release

### Moderate

Examples may include:

- Limited information disclosure
- Unsafe handling of imported or user-provided content
- Filename-handling vulnerabilities
- A flaw requiring unusual user interaction or local access

### Low

Examples may include:

- Minor hardening recommendations
- Non-sensitive error details
- Security suggestions without direct exploitability
- Issues limited to unsupported archived versions

These categories are guidelines and do not guarantee a particular response.
